# Demo Script (~8–10 min)

Goal: show that an AI generates a consumer contract from the API spec, and that contract
verification catches a breaking provider change before it would ever reach UI testing.

## Before the meeting

- `make install` once.
- Decide your generation path:
  - **Live (headline):** configure PactFlow MCP + `ANTHROPIC_API_KEY` (see README). Do a dry run.
  - **Safe:** rely on the cached path — `make generate-cached` always works offline.
- Keep the panic button ready: if a live call stalls on stage, run `make generate-cached` and continue.

## Walkthrough

### 1. The contract is generated by AI (~2 min)
> "We don't hand-write the consumer contract. We point Claude Opus at the OpenAPI spec, and — through PactFlow's MCP tools — it produces the Pact consumer test."

```bash
make generate
```

Show `consumer/src/pets.consumer.pact.test.ts`. Call out:
- the **real measured** line — `✅ Generated N lines / M scenarios in X.Xs via <mode>`;
- if MCP ran, the **PactFlow MCP tools invoked** line;
- it's matchers + provider states, not hard-coded values.

> "This is an AI draft we review — not gospel. 30 seconds of reading replaces 30–60 minutes of writing." *(State the time saved as an estimate.)*

### 2. The consumer test produces the contract (~1 min)
> "The consumer test runs against a Pact **mock** — it never touches the real API. Running it records the contract."

```bash
make consumer-test
ls consumer/pacts/
```

Open the generated pact JSON briefly — "this is the artifact a provider must satisfy."

### 3. Verify against the real provider — GREEN (~1 min)
> "Now the other side. Provider verification replays the contract against the actual service."

```bash
make verify
```

PASS. *(Optional: `make provider-up` and `curl localhost:3001/api/v1/pets/1` to show the live service is real.)*

### 4. Someone changes the API — RED (~2 min)  ← the payoff
> "A developer renames a response field from `id` to `petId`. Looks harmless. Tests on their service still pass. But our contract says otherwise."

```bash
make verify-broken
```

FAIL, with a contract-mismatch diff. Land the point:

> "That mismatch was caught at the contract layer — fast, in CI, before a single UI test ran. That's shift-left contract testing."

## Talking points

- Contract tests sit at the **right layer**: fast, focused, early feedback — far cheaper than catching this in UI/e2e.
- **AI removes the authoring cost** that usually keeps teams from adopting contract testing.
- The pact file is **provider-verifiable** and can be published to a broker for `can-i-deploy` gating (optional `--profile broker`).
